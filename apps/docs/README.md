# Conduit Documentation

## Architecture

Conduit is built as a local-first browser automation bridge. It consists of:
- **Daemon**: A local Node.js server acting as the communication hub. It listens for HTTP API requests from clients (CLI, MCP) and maintains a WebSocket connection with the browser extension.
- **Extension**: A Manifest V3 Chrome extension that executes actions within the browser environment using `chrome.tabs` and `chrome.scripting` APIs.
- **Protocol**: A shared Zod-validated schema library ensuring strict communication contracts between all components.
- **CLI / MCP**: Client interfaces that allow AI agents or human operators to interact with the daemon via REST endpoints.

## Security Model

Security is central to Conduit's design, emphasizing explicit authorization and local control:
- **Local-Only Communication**: The daemon only listens on `127.0.0.1`, preventing external network access.
- **File-Based Tokens**: Authentication relies on a cryptographically secure token generated by the daemon and stored locally in the OS-specific AppData directory (`~/.config/conduit` or `%APPDATA%\conduit`). The file has `0o600` permissions.
- **Strict Validation**: All messages passing through the system are validated using Zod schemas defined in the `@conduit/protocol` package.

## Local-First Principles

- **No Cloud Dependency**: Conduit does not require an external SaaS, cloud account, or internet connection to function.
- **Data Privacy**: All browsing data, automation steps, and snapshots stay entirely on the local machine.
- **Transparency**: Every action executed by Conduit can be audited locally.
